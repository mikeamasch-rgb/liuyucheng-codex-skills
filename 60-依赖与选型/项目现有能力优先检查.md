# 项目现有能力优先检查

这份文档用来约束一个高频问题：

还没查当前项目已有能力，就直接写新组件 / 新工具 / 新状态层。

## 每次动手前先查什么

### 1. 依赖

先查：

- `package.json`
- lockfile

看是否已有：

- 表格库
- 表单库
- 校验库
- 数据请求库
- 拖拽库
- 上传库

### 2. 组件

先查：

- `components/`
- `ui/`
- `features/`

看有没有可复用组件或半成品。

### 3. hooks / utils / services

很多时候不是没有能力，而是已经写在：

- `hooks/`
- `utils/`
- `services/`

## 检查后的决策顺序

1. 已有能力直接复用
2. 已有能力轻量扩展
3. 现有依赖可覆盖
4. 原生实现足够
5. 最后再考虑新依赖或手写

## 输出要求

开始实现前，最好写一句明确结论：

- 已检查项目现有依赖与组件
- 当前没有可直接复用能力
- 因此本次选择：
  - 复用
  - 扩展
  - 引库
  - 手写

## 反模式

不要这样做：

1. 页面里直接新写一套弹窗逻辑
2. 结果区直接新写一套表格交互
3. 没查现有服务层就又写一套 API 封装
