# Build vs Buy 技能模板

## 适用场景

当任务涉及下列能力时，先用这份模板判断，不要直接开始编码：

- 基础 UI 组件
- 表格
- 表单
- 上传
- 拖拽
- 日期选择
- 图表
- 富文本
- 认证
- 数据获取缓存

## 判断顺序

### 第一步：这是不是通用能力

如果它在很多项目都会出现，而且不是当前业务特有逻辑，默认视为通用能力。

### 第二步：项目里有没有现成东西

检查：

1. `package.json`
2. 现有组件目录
3. 现有 hooks / utils / service

### 第三步：浏览器原生够不够

如果原生能力已经能安全满足需求，不要为了“更完整”而引库。

### 第四步：成熟开源库值不值得

至少回答：

1. 它解决了什么问题
2. 当前需求是否复杂到值得引库
3. 是否与当前技术栈兼容
4. 是否明显减少维护成本

### 第五步：最后才考虑手写

只有在前四步都不成立时，才进入手写。

## 最小输出模板

每次做 build-vs-buy 判断时，输出：

1. 当前技术栈
2. 当前需求类型
3. 项目现有能力
4. 原生是否足够
5. 推荐方案
6. fallback

## 反模式

不要这样做：

1. 还没看 `package.json` 就开始写
2. 明明是复杂通用能力，却说“先写个简单版”
3. 为了少装依赖而重复造轮子
