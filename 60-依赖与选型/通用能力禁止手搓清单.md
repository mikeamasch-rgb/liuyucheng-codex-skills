# 通用能力禁止手搓清单

这份清单的目标不是绝对禁止，而是：

当出现这些需求时，默认先判断“为什么不直接复用成熟方案”，而不是默认自己写。

## 默认不要手搓的能力

### 基础交互组件

- dialog / modal
- dropdown
- popover
- tooltip
- tabs
- accordion
- combo box
- command palette

### 数据与表格

- 复杂表格
- 列宽拖拽
- 排序/筛选/分页组合表格
- 树表

### 输入与表单

- 表单状态系统
- 表单校验系统
- schema 校验器
- 日期 / 时间选择器

### 布局与交互

- resizable panels
- drag & drop
- complex split view
- floating layer system

### 数据获取与状态

- server-state cache
- query invalidation system
- polling / refetch framework

### 其他高频复杂能力

- 上传管理器
- rich text editor
- chart system
- auth framework
- PDF/export pipeline

## 默认优先顺序

遇到这类需求时，默认按这个顺序判断：

1. 项目里有没有现成实现
2. 浏览器原生是否足够
3. 成熟开源库是否更合适
4. 最后才允许最小手写

## 什么时候才允许手写

满足以下任一情况才考虑手写：

1. 需求非常小，原生能力即可
2. 项目已有实现，补一小段即可
3. 引库成本明显大于手写成本
4. 该能力不属于复杂通用能力，只是轻量业务特例

## 常见错误信号

如果出现这些想法，要先停下来：

- “这个我自己写一下应该也不难”
- “先写个简单版本，后面再说”
- “只是一个小弹窗/小下拉/小表格”

因为这类“小东西”最容易长成维护灾难。
