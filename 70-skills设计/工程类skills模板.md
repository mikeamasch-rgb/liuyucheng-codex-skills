# 工程类 Skills 模板

这份文档用于沉淀适合长期复用的“编程 / 工程”类技能模板。

目标不是做成大而全百科，而是把高频任务收成可复用 skill 结构。

## 先区分五种东西

借鉴成熟社区实践，建议严格分工：

1. `AGENTS / 常驻规则`
   - 最小常驻原则
   - 永远有效的红线

2. `Prompt / 一次性提示`
   - 某一次任务的临时上下文
   - 不应该长期沉淀成 skill

3. `Skill / 多步骤工作流`
   - 高复用
   - 有明确触发条件
   - 有稳定步骤

4. `MCP / 外部实时系统`
   - 浏览器
   - 第三方 SaaS
   - 数据库控制台
   - 文档系统

5. `Hook / 强制执行规则`
   - 格式化
   - lint
   - 提交前检查

## Skill 适合什么任务

当一个任务同时满足下面条件时，适合做 skill：

1. 高频重复
2. 多步骤
3. 有稳定结构
4. 直接靠模型临时发挥容易出错

不满足这些条件时，不要强行 skill 化。

## 推荐工程类技能清单

### 1. bug-fix

适用：

- 用户说某功能坏了
- 页面报错
- 接口失败
- 某个流程和预期不一致

最小结构：

1. 重现问题
2. 确认真实现象
3. 定位代码/日志/依赖
4. 找最小修复点
5. 验证主流程

不应该包含：

- 大规模重构
- 借修 bug 顺手改架构

---

### 2. code-review

适用：

- 用户让你 review
- 提交前风险检查

最小结构：

1. 先列问题
2. 再说影响
3. 最后补概要

重点检查：

- 行为回归
- 边界条件
- 空状态
- 错误处理
- 类型/状态不一致

---

### 3. feature-build

适用：

- 新功能开发
- 新页面 / 新接口 / 新流程

最小结构：

1. 定义目标
2. 确认主流程
3. 定义输入输出
4. 最小实现
5. 验证

重点：

- 只做 V1
- 不提前堆未来能力

---

### 4. refactor

适用：

- 文件过大
- 边界不清
- 重复明显

最小结构：

1. 明确为什么要重构
2. 列出不改什么
3. 分小步执行
4. 每步验证

重点：

- 小步
- 可回退
- 不混入功能新增

---

### 5. debug-external-api

适用：

- 外部 API 接不通
- 返回异常
- 鉴权/代理/超时问题

最小结构：

1. 验证配置
2. 验证 endpoint
3. 验证网络
4. 验证 runtime 行为
5. 验证数据落库 / 展示

重点：

- 把“代码问题”和“环境问题”拆开
- 不在没证据时断言是接口锅或代理锅

---

### 6. delivery-readiness

适用：

- 用户说“准备上线”
- 提交前收尾

最小结构：

1. 构建
2. 类型检查
3. 主流程
4. 空状态
5. 错误状态
6. 文案
7. 无关改动检查

---

## 每个工程类 Skill 建议结构

```txt
skill-name/
├─ SKILL.md
├─ references/
│  ├─ checklist.md
│  ├─ examples.md
│  └─ edge-cases.md
└─ scripts/
   └─ optional helper scripts
```

## SKILL.md 顶层模板

建议包含：

1. 触发场景
2. 不适用场景
3. 标准执行步骤
4. 何时读取 references
5. 最小验证要求

不要包含：

- 长篇背景知识
- 大量示例
- 与当前 skill 无关的组织价值观

---

## 工程类 Skill 的质量标准

一个好 skill 应该满足：

1. 另一个 agent 拿到就能执行
2. 没有大量重复通用常识
3. 触发条件清楚
4. 步骤清楚
5. 验证要求清楚
6. references 是按需而不是常驻

---

## 何时不应该写成 skill

不要把以下内容强行做成 skill：

1. 一次性项目背景
2. 某个具体任务的临时说明
3. 只有一句话就能说清的规则
4. 完全依赖实时外部状态的操作说明但没有工具支撑

这种内容更适合：

- 项目文档
- 当次 prompt
- MCP 使用说明
