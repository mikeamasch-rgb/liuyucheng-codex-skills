# 功能切分与边界控制

这份文档专门解决一个高频问题：

- 先把很多东西堆到一个页面/文件里
- 之后再被迫拆分
- 最后结构混乱、维护困难

## 核心原则

不要把“先做出来再说”当作默认实现方式。

复杂度一旦越过边界，再拆会比一开始切分更贵。

## 何时必须先切分

出现以下任一情况，就应该在编码前先切分：

1. 一个页面同时包含：
   - 搜索条件
   - 结果展示
   - 表单状态
   - API 调用
   - 复杂转换逻辑

2. 一个文件开始同时承担三类以上职责

3. 一个改动已经明显涉及：
   - UI
   - service
   - schema
   - validation
   - result mapping

## 最小切分模板

默认优先切成：

1. 页面容器
   - 只做页面组织

2. 表单组件
   - 只做输入和提交

3. 结果组件
   - 只做展示和交互

4. 服务层
   - 只做 API / 数据读写

5. 类型层
   - 只做 shared types

## 判断“是不是已经堆太多了”

如果一个文件里同时出现下面三类以上内容，通常已经该拆：

- JSX 结构
- 复杂 state
- fetch / API 调用
- 数据映射
- 校验逻辑
- 重试逻辑
- 外部依赖接入

## 页面文件建议

页面文件尽量满足：

1. 一眼能看懂页面结构
2. 不塞业务细节
3. 不塞复杂网络逻辑
4. 不塞大段数据转换

## 什么时候不要急着抽象

不要为了“以后可能有用”就提前抽象。

抽象的条件应该是：

1. 已经重复
2. 已经难改
3. 已经混职责

## 输出要求

在做新功能前，最好先写：

1. 本次会改哪些文件
2. 每个文件负责什么
3. 本次明确不做什么

如果回答不出来，说明切分还不清楚。
