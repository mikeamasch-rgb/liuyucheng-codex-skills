# 怎么使用这套规则库

这份文档回答一个实际问题：

以后做项目时，这套规则库到底怎么用，才不会又回到“长总则、乱技能、乱实现”的状态。

## 一、启动新项目时怎么做

### 第一步：给项目放一个短版 `AGENTS.md`

短版 `AGENTS.md` 只放：

1. 这个项目的目标
2. 主用户流程
3. 最小常驻规则
4. 路由到哪些专项文档

不要把所有细节重新塞进去。

参考模板：

- `10-项目模板/短版AGENTS模板.md`

### 第二步：按项目类型引用专项文档

根据项目实际需要，在短版 `AGENTS.md` 里写：

- UI 项目重点读哪份
- API 项目重点读哪份
- DB 重点读哪份

### 第三步：开始任务前先做任务分流

先判断当前任务属于：

1. UI / 页面
2. API / 后端
3. 数据库
4. 依赖选型
5. skills 设计
6. 交付验证

然后只读对应文档，不要全读。

参考：

- `80-runbooks/工程任务分流.md`

## 二、日常做项目时怎么用

### 1. 新功能

先读：

- `20-界面规则/功能切分与边界控制.md`
- `60-依赖与选型/项目现有能力优先检查.md`

如果涉及复杂通用能力，再看：

- `60-依赖与选型/build-vs-buy技能模板.md`

### 2. 页面 / 结果区 / 交互

读：

- `20-界面规则/界面与产品页规则.md`

### 3. API / 后端

读：

- `30-后端与接口/API与后端规则.md`

### 4. 数据库

读：

- `40-数据库/数据库与迁移规则.md`

### 5. 发布 / 上线前检查

读：

- `50-验证与交付/验证清单.md`
- `50-验证与交付/上线前环境一致性检查.md`

### 6. 觉得 Codex 又要乱手搓

先读：

- `60-依赖与选型/通用能力禁止手搓清单.md`
- `60-依赖与选型/build-vs-buy技能模板.md`

## 三、什么时候应该沉淀成 skill

满足以下条件时，适合沉淀成 skill：

1. 高频重复
2. 多步骤
3. 直接靠临时发挥容易出错

参考：

- `70-skills设计/skills设计规则.md`
- `70-skills设计/工程类skills模板.md`

---

## 四、不要怎么用

不要：

1. 每次都把所有文档全读一遍
2. 每个项目都复制一份超长总则
3. 把规则、示例、runbook、skill 混在同一份文档里
4. 任务还没分流就开始编码
