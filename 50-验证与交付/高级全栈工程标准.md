# 高级全栈工程标准

这份文档回答一个问题：

要把 vibecoding 做到接近高级全栈工程师标准，还差哪些能力层？

## 一、需求层

高级标准不是“听到需求就写代码”，而是先确认：

1. 用户真正想完成什么
2. 当前页面应该展示什么，不该展示什么
3. 哪些概念是内部实现，不该暴露给用户
4. 本次只做 V1 还是已经在偷做 V2/V3

## 二、切分层

高级标准要求：

1. 先切边界再写
2. 不让页面、请求、状态、转换逻辑混成一团
3. 出现复杂度苗头时就拆，不等文件爆炸后再抽

## 三、接口层

高级标准要求：

1. 输入输出清晰
2. 错误可分类
3. 页面拿到的是稳定 contract，不是随机 shape

## 四、环境层

高级标准要求：

1. 本地环境
2. dev server 运行环境
3. 构建环境
4. 生产环境

这四层要分开理解。

最常见低级错误是只在本地浏览器里验证，不验证服务端运行环境。

## 五、外部依赖层

高级标准要求：

1. 先验证第三方是否真的可达
2. 再排查 key / endpoint / 权限
3. 最后再排业务逻辑

不要一遇到失败就盲猜代码逻辑。

## 六、验证层

高级标准要求：

1. build 通过
2. 主流程实际跑一遍
3. 空状态验证
4. 错误状态验证
5. 外部 API 路径验证
6. 数据写入验证
7. 页面展示验证

## 七、表达层

高级标准要求：

1. 不把猜测说成事实
2. 不把“理论上可用”说成“已完成”
3. 发现前提错时要明确指出
4. 报告里区分：
   - 已确认
   - 未确认
   - 推测
   - 阻塞点

## 八、长期维护层

高级标准要求：

1. 规则短常驻
2. 细节按需读
3. skill 短、可复用、可维护
4. 文档不重复、不堆砌

## 九、最低判断标准

如果一个实现满足以下条件，更接近高级全栈工程标准：

1. 用户目标明确
2. 结构清晰
3. 接口稳定
4. 环境差异被考虑
5. 失败原因可追踪
6. 验证证据完整
7. 改动可维护
