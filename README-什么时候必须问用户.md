# 什么时候必须问用户

这份文档用于约束一个高频问题：

- 拿不定主意时不能硬猜
- 但也不能遇到一点小事就打断用户

## 一、必须问用户的情况

### 1. 产品目标不明确

例如：

- 用户到底要单页还是多页
- 结果区是要管理型还是展示型
- 某个内部概念是否应该暴露

### 2. 多种方案差异很大

例如：

- 是引入成熟库还是继续保持零依赖
- 是保留旧页面还是彻底删除
- 是做临时 fallback 还是做长期结构升级

### 3. 会影响长期结构

例如：

- 改架构
- 改目录结构
- 改数据模型
- 改路由模型

### 4. 需要用户提供外部事实

例如：

- API key
- 代理端口
- 线上环境配置
- 第三方系统信息

### 5. 会影响用户明确偏好

例如：

- 中文还是英文
- 单页还是多页
- UI 风格取舍

## 二、不需要问用户的情况

默认可以自己做：

1. 小范围代码整理
2. 已确认方向下的最小修复
3. 明确无争议的验证动作
4. 已有规则明确指向的选型动作

## 三、推荐提问方式

提问不要发散，尽量收成：

1. 现在有哪两个真实选项
2. 每个选项的代价是什么
3. 推荐哪个
4. 为什么

## 四、避免两种错误

### 错误 1：什么都不问，硬猜

后果：

- 方向偏
- 做完返工

### 错误 2：什么都问，打断过多

后果：

- 节奏慢
- 小事也停住

最好的方式是：

只在“方向性决策”时问，
不是在“执行细节”时问。
